# Internationalisation (i18n)

## Overview

The application uses **hapi-i18n**, a Hapi-native plugin, for English (`en`) and Welsh (`cy`) translations. Language preference is resolved in this order:

1. URL path prefix — `/en/` or `/cy/`
2. Query parameter — `?lang=en` or `?lang=cy`
3. Cookie — stored preference from a previous visit
4. `Accept-Language` header — browser preference
5. Default — English (`en`)

---

## Project structure

```
src/
├── config/
│   └── i18n.js                                  # hapi-i18n configuration
├── locales/
│   ├── en/translation.json                       # English translations
│   └── cy/translation.json                       # Welsh translations
└── server/
    └── common/
        └── components/language-switcher/         # Language toggle component
```

---

## Configuration

The plugin is configured in `src/config/i18n.js`:

```javascript
{
  localesPath: path.resolve('src/locales'),
  locale: 'en',                 // default language
  fallbacks: { default: 'en' }, // fallback if a key is missing
  updateFiles: false,           // do not auto-create missing keys
  extension: '.json'
}
```

## Adding translations

### 1. Add keys to both translation files

**`src/locales/en/translation.json`:**

```json
{
  "pages": {
    "home": {
      "title": "Welcome to EPR",
      "description": "Register your organisation..."
    }
  }
}
```

**`src/locales/cy/translation.json`:**

```json
{
  "pages": {
    "home": {
      "title": "Croeso i EPR",
      "description": "Cofrestrwch eich sefydliad..."
    }
  }
}
```

Use dot notation to organise keys. Common top-level groupings already in use:

```
header.serviceName
pages.home.title
navigation.about
common.yes
languageSwitcher.english
```

### 2. Use translations in templates

```nunjucks
<h1>{{ t('pages.home.title') }}</h1>
<p>{{ t('pages.home.description') }}</p>
```

### 3. Use translations in controllers

```javascript
export const myController = {
  handler(request, h) {
    return h.view('mypage', {
      pageTitle: request.t('pages.home.title')
    })
  }
}
```

To get the current locale in a controller:

```javascript
const locale = request.getLocale() // 'en' or 'cy'
```

## Language switcher

Include the language switcher component in your layout or template:

```nunjucks
{% include "language-switcher/index.njk" %}
```

The component renders a link to the same page in the other language. It is styled via `src/client/stylesheets/components/_language-switcher.scss`.

## URL patterns

The application supports language-prefixed URLs for all content routes:

| Language | URL pattern        |
| -------- | ------------------ |
| English  | `/` or `/en/about` |
| Welsh    | `/cy/about`        |

You can also switch language on any page using the `?lang=` query parameter:

```
/about?lang=cy   # Welsh
/about?lang=en   # English
```

## Adding language support to new routes

Register both a default path and a language-prefixed path for every new page:

```javascript
// src/server/mypage/index.js
import { mypageController } from './controller.js'

export const mypage = {
  plugin: {
    name: 'mypage',
    register(server) {
      server.route([
        {
          method: 'GET',
          path: '/mypage',
          ...mypageController
        },
        {
          method: 'GET',
          path: '/{language}/mypage',
          ...mypageController
        }
      ])
    }
  }
}
```

## Interpolation

Use `{{variable}}` placeholders in translation values:

**Translation file:**

```json
{
  "greeting": "Hello {{name}}, welcome to {{service}}"
}
```

**Template:**

```nunjucks
{{ t('greeting', { name: userName, service: 'EPR' }) }}
```

**Controller:**

```javascript
request.t('greeting', { name: userName, service: 'EPR' })
```

---

## Pluralisation

Use nested `one` / `other` keys:

**Translation file:**

```json
{
  "items": {
    "one": "You have 1 item",
    "other": "You have {{count}} items"
  }
}
```

The correct form is selected automatically based on the `count` value passed as an interpolation variable.

## Best practices

- Group related keys under a common parent (e.g. `pages.home.*`)
- Use camelCase for key names: `serviceName` not `service-name`
- Add translations to **both** language files at the same time
- Keep a glossary of agreed Welsh equivalents for technical terms
- Test the layout in both languages — Welsh strings are often longer

## Troubleshooting

### Translation key not found / shows raw key

1. Check the key exists in both `src/locales/en/translation.json` and `src/locales/cy/translation.json`
2. Check for typos in the dot-notation path
3. Verify the template syntax: `{{ t('key') }}`
4. Restart the development server

### Language not changing

1. Verify the URL prefix is `/en/` or `/cy/`
2. Check that the route registers both a default path and a `/{language}/…` path
3. Try the query parameter: `?lang=cy`

## Migration from i18next

The application previously used i18next. If you encounter old syntax:

| Context    | Old (i18next)                         | New (hapi-i18n)    |
| ---------- | ------------------------------------- | ------------------ |
| Template   | `{{ 'key' \| t(currentLanguage) }}`   | `{{ t('key') }}`   |
| Controller | `i18next.t('key', { lng: language })` | `request.t('key')` |
